Lazy Loading

Don't load stuff until you actually need it — faster apps, less bandwidth, happier users

JavaScript React Performance

The Restaurant Analogy

Imagine you walk into a restaurant. There are two ways they could serve you:

Without lazy loading

The kitchen cooks every single dish on the menu the moment you sit down. You only ordered a salad, but they also made 47 pasta dishes, 12 steaks, and a birthday cake. Your salad takes forever because the kitchen is swamped doing work nobody asked for.

With lazy loading

The kitchen only cooks your salad when you order it. If you later want dessert, they'll make it then. Fast, efficient, no wasted work.

Core principle: Lazy loading = don't load stuff until you actually need it. Your app starts faster because it only downloads code (or images, or data) the user actually uses, right when they use it.

Eager vs Lazy: Static import

Normally, when you import something at the top of a file, it gets loaded immediately when the page loads — even if the user never sees it. These are called static imports (or "eager" imports).

TypeScript// These are loaded IMMEDIATELY when this file runs — no matter what
import { Chart } from 'chart.js';
import { PDFExporter } from './pdf-exporter';
import { MarkdownEditor } from './markdown-editor';
import { VideoPlayer } from './video-player';

Even if the user never opens the chart, never exports a PDF, never writes markdown, and never plays a video — all that code is downloaded and parsed upfront. That's a lot of wasted work.

Dynamic import()

JavaScript has a built-in solution: the dynamic import() expression. Instead of loading a module at the top of the file, you call import() as a function inside your code. It returns a Promise that resolves to the module when it's ready.

TypeScript// Nothing is loaded yet — this just defines a function
async function openChartPanel() {
  // NOW the chart code downloads — only when the user clicks "Show Chart"
  const { Chart } = await import('chart.js');

  const chart = new Chart(canvas, {
    type: 'bar',
    data: salesData,
  });
}

The key difference:

Static import (eager)

import { Chart } from 'chart.js';

Loaded immediately at page startup. Blocks rendering until downloaded and parsed.

Dynamic import (lazy)

const { Chart } = await import('chart.js');

Loaded on-demand when this line executes. The rest of the page loads instantly.

How bundlers handle this

When tools like Webpack or Vite see a dynamic import(), they automatically split that module into a separate chunk (a separate .js file). At runtime, the browser only fetches that chunk when the import() is called. This is called code splitting.

Practical Example: Settings Page

Imagine a dashboard app with a settings page. The settings page has tabs: Profile, Notifications, Billing, and Admin (only for admins). Each tab has complex UI. Without lazy loading, every user downloads the code for every tab — including the Admin panel they'll never see.

TypeScript// settings-page.ts

type TabName = 'profile' | 'notifications' | 'billing' | 'admin';

async function loadTab(tab: TabName) {
  switch (tab) {
    case 'profile':
      return import('./tabs/profile');
    case 'notifications':
      return import('./tabs/notifications');
    case 'billing':
      return import('./tabs/billing');
    case 'admin':
      return import('./tabs/admin');
  }
}

// When user clicks a tab:
tabButton.addEventListener('click', async () => {
  const module = await loadTab('billing');
  module.render(container);
});

Each tab's code lives in its own file. The bundler creates separate chunks. Users only download the tab they click on.

React: React.lazy + Suspense

React has a built-in pattern for lazy loading components. You wrap a dynamic import() with React.lazy(), and React handles the rest — including showing a fallback while the component downloads.

Basic pattern

TSXimport React, { Suspense, lazy } from 'react';

// This does NOT load the chart code yet — just creates a lazy wrapper
const ChartPanel = lazy(() => import('./ChartPanel'));

function Dashboard() {
  const [showChart, setShowChart] = useState(false);

  return (
    <div>
      <button onClick={() => setShowChart(true)}>Show Chart</button>

      {showChart && (
        <Suspense fallback={<p>Loading chart...</p>}>
          <ChartPanel />
        </Suspense>
      )}
    </div>
  );
}

What happens step by step:

  1. React.lazy() wraps the dynamic import() — the chart code is not loaded yet
  2. The user clicks "Show Chart" and showChart becomes true
  3. React tries to render <ChartPanel /> for the first time, triggering the download
  4. While downloading, <Suspense fallback={...}> renders "Loading chart..." as a placeholder
  5. Once the code arrives, the real <ChartPanel> renders and replaces the fallback

Skeleton placeholders

Instead of showing text like "Loading...", you can show a skeleton that mimics the shape of the real content. This feels faster to users because they see the layout immediately.

TSXconst UserProfile = lazy(() => import('./UserProfile'));

function ProfileSkeleton() {
  return (
    <div className="profile-skeleton">
      <div className="skeleton-avatar" />
      <div className="skeleton-line wide" />
      <div className="skeleton-line" />
    </div>
  );
}

// Skeleton shows instantly, real profile swaps in when ready
<Suspense fallback={<ProfileSkeleton />}>
  <UserProfile userId={42} />
</Suspense>
Fallback options: Use fallback={null} to show nothing (invisible loading), a spinner for generic loading states, or a skeleton component for the best perceived performance. Match the skeleton's shape to the real component's layout.

Route-Based Code Splitting

The most impactful place to lazy load is at the route level. Each page of your app becomes a separate chunk that only downloads when the user navigates to it.

TSXimport { BrowserRouter, Routes, Route } from 'react-router-dom';
import React, { Suspense, lazy } from 'react';

// Each page is a lazy-loaded chunk
const Home       = lazy(() => import('./pages/Home'));
const Dashboard  = lazy(() => import('./pages/Dashboard'));
const Settings   = lazy(() => import('./pages/Settings'));
const AdminPanel = lazy(() => import('./pages/AdminPanel'));

function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<PageSpinner />}>
        <Routes>
          <Route path="/"         element={<Home />} />
          <Route path="/dashboard" element={<Dashboard />} />
          <Route path="/settings"  element={<Settings />} />
          <Route path="/admin"     element={<AdminPanel />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}

A user who only visits the Home and Dashboard pages never downloads the Settings or Admin code at all.

Caching with memoizePromise

Once a lazy module is loaded, you don't want to re-download it on the second call. A common pattern is to wrap the import() in a memoiser that caches the promise:

TypeScriptfunction memoizePromise<T>(factory: () => Promise<T>): () => Promise<T> {
  let cached: Promise<T> | null = null;
  return () => {
    if (!cached) {
      cached = factory();
    }
    return cached;
  };
}

// Usage: the PDF exporter only downloads once, no matter how many times called
const getPdfExporter = memoizePromise(() => import('./pdf-exporter'));

exportButton.addEventListener('click', async () => {
  const { exportToPdf } = await getPdfExporter();
  exportToPdf(document);
});

Good news

React.lazy() and native import() already cache internally — the browser won't re-fetch a module it already has. The memoizePromise pattern is useful when you want to control caching yourself, e.g. in non-React code or when you need to pass the loaded module around.

Image Lazy Loading

Lazy loading isn't just for JavaScript. Images are often the heaviest assets on a page, and modern browsers have built-in support for deferring them.

The loading="lazy" attribute

HTML<!-- This image only loads when it enters (or nears) the viewport -->
<img
  src="hero-photo.jpg"
  alt="Mountain landscape"
  loading="lazy"
  width="800"
  height="400"
/>

<!-- Compare: this image loads immediately (default behaviour) -->
<img
  src="logo.png"
  alt="Site logo"
  loading="eager"
/>

Don't lazy load above-the-fold images

Images visible on initial page load (hero images, logos) should use loading="eager" (the default). Lazy loading them would make the page feel slower because the user sees a blank space before the image downloads.

Intersection Observer (custom control)

For more control — like triggering animations or loading data when an element scrolls into view — use the IntersectionObserver API:

TypeScriptconst observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      const img = entry.target as HTMLImageElement;
      img.src = img.dataset.src!;
      img.classList.add('loaded');
      observer.unobserve(img);
    }
  });
}, { rootMargin: '200px' }); // Start loading 200px before it's visible

document.querySelectorAll('img[data-src]').forEach((img) => {
  observer.observe(img);
});

When NOT to Lazy Load

Scenario Why Not
Small modules (<5KB) The HTTP request overhead costs more than just bundling it. The round-trip to fetch a tiny file can be slower than including it upfront.
Critical-path UI Anything the user sees on first render (navigation, hero section, main content) should load eagerly. Lazy loading it creates a visible flicker.
Above-the-fold images The logo and hero image should be loading="eager". Deferring them makes the initial paint feel broken.
Frequently used features If 95% of users use a feature on every visit, eager loading is simpler and avoids the loading state.

Why Bother?

Real-world apps are massive. If they loaded every feature's code upfront — charts, PDF export, admin panels, video players, settings — the initial page load would be painfully slow. Lazy loading gives you:

  • Faster startup — only load the core shell and the current page
  • Less bandwidth — users who never open settings never download its code
  • Better perceived performance — show a placeholder instantly, swap in the real content when ready
  • Smaller initial bundle — the JavaScript the browser needs to parse on first load shrinks dramatically
TL;DR: Lazy loading = "I'll deal with it when I need to." Your app loads faster because it only downloads code the user actually uses, right when they use it.
