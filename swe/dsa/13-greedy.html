Greedy

Make the locally optimal choice at each step — and sometimes that's all you need for a globally optimal solution

TypeScript Optimisation

What is a Greedy Algorithm?

A greedy algorithm builds a solution piece by piece, always choosing the option that looks best right now without reconsidering earlier choices. Unlike dynamic programming, which explores all possibilities and picks the global optimum, greedy commits to each decision and never backtracks.

For a greedy approach to produce an optimal solution, the problem must satisfy two properties:

  1. Greedy choice property: A globally optimal solution can be arrived at by making a locally optimal choice. Picking the best option at each step doesn't lock us out of the overall best answer.
  2. Optimal substructure: An optimal solution to the problem contains optimal solutions to its subproblems — the same property DP requires.

When both properties hold, greedy is strictly better than DP: same result, simpler code, usually faster. The difficulty lies in proving greedy works for a given problem.

When Does Greedy Work?

Greedy algorithms work when the problem has a matroid structure — roughly, when you can safely commit to local decisions. In practice, you develop intuition by studying classic problems and recognising patterns.

Greedy vs DP: Both require optimal substructure. The difference is that DP problems have overlapping subproblems where the greedy choice property fails — you need to consider multiple branches. Greedy problems let you prune to a single branch at each step.

Rule of thumb: If you can sort the input and make a single pass with a simple invariant, it's probably greedy. If you need to "try both taking and skipping" an element, it's DP.

Jump Game

Given an array where each element represents the maximum jump length from that position, determine if you can reach the last index. The greedy insight: track the farthest reachable index as you scan left to right. If you ever land on a position beyond your reach, you're stuck.

function canJump(nums: number[]): boolean {
  let maxReach = 0;

  for (let i = 0; i < nums.length; i++) {
    if (i > maxReach) return false;
    maxReach = Math.max(maxReach, i + nums[i]);
  }

  return true;
}

// canJump([2, 3, 1, 1, 4])  → true   (0→1→4 or 0→2→3→4)
// canJump([3, 2, 1, 0, 4])  → false  (stuck at index 3)

Why greedy works: We never need to decide which path to take — we only care whether the destination is reachable at all. Tracking the running maximum is enough because any position within reach can be visited.

Jump Game II

Same setup, but now find the minimum number of jumps to reach the end. Think of it as BFS over reachable ranges: each "level" is one jump, and we track the farthest position reachable in the current and next levels.

function jump(nums: number[]): number {
  let jumps = 0;
  let currentEnd = 0;
  let farthest = 0;

  for (let i = 0; i < nums.length - 1; i++) {
    farthest = Math.max(farthest, i + nums[i]);

    if (i === currentEnd) {
      jumps++;
      currentEnd = farthest;
    }
  }

  return jumps;
}

// jump([2, 3, 1, 1, 4])  → 2  (0→1→4)
// jump([2, 3, 0, 1, 4])  → 2  (0→1→4)

BFS analogy

currentEnd marks the boundary of the current BFS level. When we reach it, we "jump" to the next level whose boundary is farthest. Each level transition costs one jump. This guarantees the minimum because BFS explores all positions at distance k before any at distance k+1.

Gas Station

There are n gas stations on a circular route. Station i has gas[i] fuel and costs cost[i] to reach the next station. Find the starting station index that lets you complete the circuit, or return -1 if impossible.

Key insight: If the total gas ≥ total cost, a solution must exist. To find the starting point, track the running surplus. Whenever it goes negative, the start must be after the current station — reset and try the next one.

function canCompleteCircuit(
  gas: number[],
  cost: number[]
): number {
  let totalSurplus = 0;
  let currentSurplus = 0;
  let start = 0;

  for (let i = 0; i < gas.length; i++) {
    const diff = gas[i] - cost[i];
    totalSurplus += diff;
    currentSurplus += diff;

    if (currentSurplus < 0) {
      start = i + 1;
      currentSurplus = 0;
    }
  }

  return totalSurplus >= 0 ? start : -1;
}

// canCompleteCircuit([1,2,3,4,5], [3,4,5,1,2])  → 3
// canCompleteCircuit([2,3,4], [3,4,3])           → -1
Why one pass works: If we fail at station j starting from s, then every station between s and j would also fail (they'd have even less fuel). So we can safely skip ahead to j+1 as the new candidate start.

Hand of Straights

Given an array of integers (cards) and a group size k, determine if the cards can be rearranged into groups of k consecutive numbers. Greedy approach: always start a group from the smallest available card.

function isNStraightHand(
  hand: number[],
  groupSize: number
): boolean {
  if (hand.length % groupSize !== 0) return false;

  const counts = new Map<number, number>();
  for (const card of hand) {
    counts.set(card, (counts.get(card) ?? 0) + 1);
  }

  const sorted = [...counts.keys()].sort((a, b) => a - b);

  for (const start of sorted) {
    const count = counts.get(start)!;
    if (count === 0) continue;

    for (let j = 0; j < groupSize; j++) {
      const card = start + j;
      const available = counts.get(card) ?? 0;
      if (available < count) return false;
      counts.set(card, available - count);
    }
  }

  return true;
}

// isNStraightHand([1,2,3,6,2,3,4,7,8], 3)  → true  ([1,2,3],[2,3,4],[6,7,8])
// isNStraightHand([1,2,3,4,5], 4)           → false

Why greedy works: The smallest card must be the start of some group — it can't be in the middle or end of a consecutive sequence because there's nothing smaller to start it. So we greedily build a group from it and repeat.

Task Scheduler

Given tasks with a cooldown period n between identical tasks, find the minimum time to execute all tasks. The insight: the most frequent task dictates the structure. Place the most frequent tasks first, then fill the gaps.

function leastInterval(
  tasks: string[],
  n: number
): number {
  const freq = new Array(26).fill(0);
  for (const task of tasks) {
    freq[task.charCodeAt(0) - 65]++;
  }

  const maxFreq = Math.max(...freq);
  const maxCount = freq.filter(f => f === maxFreq).length;

  // (maxFreq - 1) full chunks of size (n + 1), plus final partial chunk
  const minSlots = (maxFreq - 1) * (n + 1) + maxCount;

  return Math.max(minSlots, tasks.length);
}

// leastInterval(["A","A","A","B","B","B"], 2)  → 8
// Schedule: A B _ A B _ A B  (8 slots)
//
// leastInterval(["A","A","A","B","B","B"], 0)  → 6
// Schedule: A B A B A B  (no idle needed)

Visualising the formula

Imagine a grid with maxFreq rows and n+1 columns. Place the most frequent task in column 0 of every row. Fill remaining columns with other tasks. The idle slots are whatever gaps remain. If there are enough tasks to fill all gaps, the answer is simply tasks.length.

When Greedy Fails

Counterexample: Coin Change

With standard denominations (1, 5, 10, 25), greedy works: always pick the largest coin that fits. But with non-standard denominations, it fails.

Example: Coins = [1, 3, 4], amount = 6.

  • Greedy: Pick 4, then 1, then 1 → 3 coins (4 + 1 + 1).
  • Optimal: Pick 3, then 3 → 2 coins (3 + 3).

The greedy choice (pick the biggest coin) doesn't have the greedy choice property here. You need DP to explore both "take 4" and "take 3" branches. This is why coin change is a classic DP problem.
