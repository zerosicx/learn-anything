Tries

Prefix trees for lightning-fast string operations — autocomplete, spell check, and word search in O(m) time

TypeScript Strings

What is a Trie?

A trie (pronounced "try", from retrieval) is a tree-like data structure where each node represents a single character. Paths from the root to marked nodes spell out stored words. Unlike a hash map that stores complete strings, a trie shares common prefixes — the word "car" and "card" share the path c → a → r.

This prefix-sharing property makes tries ideal for:

  • Autocomplete: Given a prefix, find all matching words by traversing to the prefix node and collecting descendants.
  • Spell checking: Check if a word exists in a dictionary with O(m) lookups where m is word length.
  • IP routing: Longest prefix matching in network routers.
  • Word games: Efficiently validate words during board traversal (Boggle, Scrabble).
Trie vs HashMap for strings: A HashMap can check if a word exists in O(m) time too. The trie's advantage is prefix operations — finding all words with a given prefix is O(p + k) where p is prefix length and k is the number of results. With a HashMap, you'd need to iterate every stored key.

Trie Node Implementation

TrieNode and Trie

class TrieNode {
  children: Map<string, TrieNode> = new Map();
  isEnd = false;
}

class Trie {
  private root = new TrieNode();

  insert(word: string): void {
    let node = this.root;
    for (const char of word) {
      if (!node.children.has(char)) {
        node.children.set(char, new TrieNode());
      }
      node = node.children.get(char)!;
    }
    node.isEnd = true;
  }

  search(word: string): boolean {
    const node = this._traverse(word);
    return node !== null && node.isEnd;
  }

  startsWith(prefix: string): boolean {
    return this._traverse(prefix) !== null;
  }

  private _traverse(str: string): TrieNode | null {
    let node = this.root;
    for (const char of str) {
      if (!node.children.has(char)) return null;
      node = node.children.get(char)!;
    }
    return node;
  }
}

Map vs Array for children

Using a Map<string, TrieNode> is flexible — it handles any character set. For lowercase English only, you could use a fixed-size array of 26 slots (children[char.charCodeAt(0) - 97]) for slightly better cache performance. In interviews, either is acceptable.

Word Search in a Trie

A common extension: support . as a wildcard matching any single character. The search method becomes recursive — when encountering a dot, try all children.

class WordDictionary {
  private root = new TrieNode();

  addWord(word: string): void {
    let node = this.root;
    for (const char of word) {
      if (!node.children.has(char)) {
        node.children.set(char, new TrieNode());
      }
      node = node.children.get(char)!;
    }
    node.isEnd = true;
  }

  search(word: string): boolean {
    return this._search(word, 0, this.root);
  }

  private _search(word: string, idx: number, node: TrieNode): boolean {
    if (idx === word.length) return node.isEnd;

    const char = word[idx];

    if (char === ".") {
      // Wildcard: try every child
      for (const child of node.children.values()) {
        if (this._search(word, idx + 1, child)) return true;
      }
      return false;
    }

    const next = node.children.get(char);
    if (!next) return false;
    return this._search(word, idx + 1, next);
  }
}

// const dict = new WordDictionary();
// dict.addWord("bad");  dict.addWord("dad");  dict.addWord("mad");
// dict.search("pad")  → false
// dict.search(".ad")  → true  (matches bad, dad, mad)
// dict.search("b..")  → true  (matches bad)

Wildcard Complexity

Without wildcards, trie search is O(m). With wildcards, the worst case is O(26^m) for a word of all dots — every path must be explored. In practice, most patterns have few dots, so performance is much better than the worst case.

Autocomplete with a Trie

Given a prefix, return all words in the trie that start with that prefix. Navigate to the prefix node, then do a DFS collecting all words from there.

class AutocompleteTrie extends Trie {
  private root = new TrieNode();

  insert(word: string): void {
    let node = this.root;
    for (const char of word) {
      if (!node.children.has(char)) {
        node.children.set(char, new TrieNode());
      }
      node = node.children.get(char)!;
    }
    node.isEnd = true;
  }

  autocomplete(prefix: string): string[] {
    let node = this.root;
    for (const char of prefix) {
      if (!node.children.has(char)) return [];
      node = node.children.get(char)!;
    }

    const results: string[] = [];
    this._collect(node, prefix, results);
    return results;
  }

  private _collect(
    node: TrieNode,
    path: string,
    results: string[]
  ): void {
    if (node.isEnd) results.push(path);

    for (const [char, child] of node.children) {
      this._collect(child, path + char, results);
    }
  }
}

// const trie = new AutocompleteTrie();
// ["cat","car","card","care","do","dog"].forEach(w => trie.insert(w));
// trie.autocomplete("car")  → ["car", "card", "care"]
// trie.autocomplete("do")   → ["do", "dog"]
// trie.autocomplete("b")    → []
The collect pattern: Navigating to the prefix node takes O(p). Then DFS from that node visits every descendant, taking O(k) where k is the total characters in all matching words. This is dramatically faster than scanning an entire dictionary.

Complexity Analysis

How does a trie compare with other data structures for common string operations? Let m = word length, n = number of stored words, p = prefix length.

Operation Trie HashMap Sorted Array
Insert O(m) O(m) O(n log n)
Search exact word O(m) O(m) O(m log n)
Prefix check O(p) O(n·m) O(m log n)
All words with prefix O(p + k) O(n·m) O(m log n + k)
Wildcard search O(26^m)* O(n·m) O(n·m)
Space O(n·m·Σ) O(n·m) O(n·m)

* Worst-case; practical performance is much better. Σ = alphabet size (26 for lowercase English). k = total chars in matching results.

When NOT to use a Trie

Tries use more memory than hash maps due to per-node overhead. If you only need exact lookups (no prefix operations), a HashMap is simpler and more memory-efficient. Use a trie when prefix-based queries are a core requirement.
